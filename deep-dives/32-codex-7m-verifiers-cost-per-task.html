Codex 700 萬用戶、Verifiers v1 同「Cost per Task」新範式——AI 編碼嘅地殼變動

📅 2026-07-14 🎙 AINews 新聞摘要 👥 主編 swyx + Alessio Fanelli 🏷 AI編碼 Codex Verifiers CostPerTask GrokBuild

呢個星期一嘅 AINews 係 2026 年 7 月 14 號播出，由 swyx 同 Alessio Fanelli 主持。這期嘅主題異常聚焦——AI 編碼領域正經歷一場地殼變動：OpenAI Codex 用戶量喺短短 24.5 小時內從 600 萬爆升到 700 萬，Prime Intellect 開源咗 Verifiers v1 架構，仲有 Grok Build 嘅隱私風波。但真正令整個行業感到震動嘅，係一個新嘅評估範式正在悄然形成——「Cost per Task」。呢個指標如果站穩腳，將會摧毀我哋對模型比較嘅所有既有認知。

以下係我對成集節目嘅深度摘要同分析，覆蓋七個核心話題加上一個整合性嘅洞察。

一、Codex 嘅 10x 增長故事

呢個禮拜嘅最大新聞毫無疑問係 OpenAI Codex 嘅用戶增長曲線。Alessio 開場就用一種「你唔會信」嘅語氣報告咗呢啲數字：

  • Jan 2026：約 550k–700k 用戶。當時已經係業界最大嘅 coding agent 平台之一，但仲未引起大眾注意
  • Mar 2026：約 200 萬 用戶。兩個月翻三倍，曲線開始陡峭
  • Jul 12（週日傍晚）：600 萬 用戶。曲線已經變成垂直
  • Jul 13（週一傍晚）：700 萬 用戶。24.5 小時內淨增 100 萬 用戶

呢個意味住由年初到而家大約 6.5 個月，Codex 用戶量增長咗 10–12x。呢種增長率喺任何 SaaS 產品都罕見，更何況係一個單一功能嘅 coding agent。

「Seven million users. That's insane! That's like — that's like being a country. A country of developers.」 — swyx

增長加速嘅引擎：

swyx 同 Alessio 討論咗幾個催化劑。最直接嘅係 GPT-5.6 Sol 發佈（Jul 9）。Sol 嘅最強能力恰好體現喺編碼同長上下文處理——Codex 作為 GPT-5.6 嘅旗艦應用，直接食到版本升級嘅紅利。

但更重要嘅係 OpenAI 取消咗 Codex 嘅用量限制，並提供一次「banked reset」——即係用戶之前未用嘅配額可以重新啟用。呢個轉變嘅策略意義巨大：OpenAI 由「限量高價」轉向「無限用量搶佔市場」，目標好明顯係用規模打敗 Claude Code。

「They just said 'go nuts'. And the market said 'okay'.」 — Alessio Fanelli

值得留意嘅係，Codex 增長主要來自口碑傳播，而非傳統廣告。700 萬開發者中，絕大部分係通過 Twitter/X 、YouTube 同 Discord 社群自然發現。呢種病毒式增長意味住產品-market fit 極強，強到足以淹沒任何競爭對手嘅市場推廣努力。

swyx 將 Codex 比作「開發者嘅 ChatGPT 時刻」——正如 2023 年 ChatGPT 用兩個月達到 1 億用戶，Codex 喺 coding 垂直領域重現咗同樣嘅爆炸式增長，但起點更高、曲線更陡。

二、Claude Code vs Codex——統計口徑嘅戰爭

咁強勁嘅 Codex 增長，對競爭對手意味住咩？swyx 同 Alessio 深入探討咗 Anthropic 嘅處境。

Anthropic 最後一次披露 Claude Code 用戶數係 2026 年 2 月：約 200 萬 用戶，年化收入（ARR）約 $25 億。五個月前呢個數字仲睇起嚟好勁——但對比 Codex 嘅 700 萬，Claude Code 已經明顯落後。

Claude Code 嘅結構性調整：

swyx 指出，Anthropic 將主要編碼能力 遷移至 Claude Tag——一個 Slackbot 形態嘅產品。呢個決定意味住 Claude Code 嘅「純 IDE agent」定位正在被淡化，取而代之嘅係嵌入工作流平台嘅方案。呢種策略有助於企業採用，因為 Slack 係好多公司嘅日常工作核心，但不利於同 Codex 呢種「開發者專用工具」正面競爭。

「Claude Tag is the new Claude Code. But 'Slackbot for coding' is a different story from 'IDE-native coding agent with 7M users'.」 — swyx

Fable 5 訂閱嘅「驚喜延期」：

另一個引發猜測嘅事件係 Claude Fable 5 訂閱計劃延期。Anthropic 嘅 Fable 訂閱原本預計喺 7 月初全面推出，但喺最後一刻推遲，官方理由係「基礎設施容量問題」。外界普遍猜測呢同 Codex 壓力有關——OpenAI 取消用量限制後，Anthropic 被迫重新評估自身嘅定價同容量策略。

Alessio 提出一個更技術性嘅觀察：推理優化。Anthropic 近期對 Fable 系列進行咗推理效率升級，帶嚟大約 10% 嘅更多可用用量。呢項優化令 Fable 5 實際上比發布初期更平、更快——但對比 Codex 嘅「無限用量」模式，呢 10% 嘅改善遠遠唔夠。

swyx 嘅總結頗為冷靜：Anthropic 唔會就此消失，但佢哋必須喺未來 3–6 個月內在 coding agent 領域做出重大策略調整。單純嘅模型質量優勢已經唔足夠——Codex 用規模同用戶體驗建立咗一個飛輪。

三、Prime Intellect Verifiers v1——開源 RL 架構嘅里程碑

如果話 Codex 嘅故事係「封閉產品如何征服市場」，咁 Prime Intellect 嘅 Verifiers v1 就係「開源基礎設施如何重新定義開發」。呢個由 Prime Intellect 發佈嘅開源強化學習（RL）架構，被 swyx 稱為「2026 年最重要嘅開源 infra 項目之一」。

環境棧重構：taskset, harness, runtime 三大抽象

Prime Intellect 將複雜嘅 RL 訓練流程拆成三個清晰嘅抽象層：

  • taskset： 定義任務嘅輸入/輸出規範，類似一個 type-safe 嘅 API 合約
  • harness： 連接觸發器同執行引擎嘅中間層，負責 rollout 管理同 reward 計算
  • runtime： 實際運行 agent 嘅執行環境，支援 vLLM 等推理引擎

呢個三層設計嘅價值在於 「bring your own harness」——你可以自帶 harness 插入 prime intellect 嘅框架，無需重新實現成個 RL pipeline。呢個設計直接支援 coding agent 同 computer-use agent 嘅異構運行。

「The beauty of Verifiers v1 is that you don't have to rebuild the entire RL stack just to swap out the environment. You bring your harness, they bring the distributed training.」 — swyx

核心技術突破：message DAG

Verifiers v1 最關鍵嘅技術創新係 message DAG（有向無環圖） 嚟存儲 rollout traces。呢個改變解決咗 RL 訓練中一個長期存在嘅性能瓶頸：

傳統 rollout trace 存儲係線性嘅——每一步嘅 token 都前後依賴，導致 verify 步驟需要 O(n²) 嘅複雜度。Message DAG 通過將獨立步驟並行化，將 verify 複雜度降至 O(n)——對於長 rollout（例如 40 輪 SWE-bench 場景），呢個係巨大嘅加速。

實測數據：

Prime Intellect 公開咗一組令人印象深刻嘅基準測試：

  • 模型：100B 參數嘅 reasoning model
  • 任務：40 輪 SWE-bench verification
  • RL steps：1000
  • 硬件：6 台 H200
  • 完成時間：少於 2 天

呢個數據對比傳統嘅 RL 訓練方案——通常需要數十台 GPU、數週時間——係一個數量級嘅改善。

技術細節：vLLM 整合與 tokenization drift 防護

Verifiers v1 深度整合 vLLM 做推理引擎，同時引入一個好聰明嘅防護機制：exact token IDs。呢個解決嘅問題係：當同一個 prompt 通過唔同嘅 tokenizer 版本或設置時，tokenization 會 drift，導致訓練同推理之間嘅不一致。Exact token IDs 確保 training-time 同 inference-time 嘅 tokenization 完全一致，避免咗呢種 drift 引入嘅噪聲。

Prime Intellect 自身嘅市場信號

swyx 順帶提到，Prime Intellect 目前估值約 $10 億（獨角獸），年化收入（ARR）約 $1 億。呢個數字反映咗開發者基礎設施市場嘅巨大需求——特別係開源 RL 框架呢個仲未被充分服務嘅領域。

Alessio 補充話，Verifiers v1 嘅發佈時間點並非巧合：OpenAI 同 Anthropic 都喺內部大規模使用 RL 訓練 coding agent，但呢啲工具都係封閉嘅。Prime Intellect 嘅開源策略，係希望成為「RL 訓練嘅 Linux」——一個任何人都可以自建嘅基礎層。

四、Harness 即產品——競爭新格局

呢個觀點喺節目中由 swyx 引用咗知名開發者 threepointone 嘅一句話：

「The harness is the app.」 — threepointone

呢句話捕捉咗 Verifiers v1 架構設計背後嘅更深層邏輯：喺模型商品化嘅時代，競爭嘅關鍵不再是模型本身，而是圍繞模型嘅「harness」——即係連接模型同實際任務嘅基礎設施。

LangChain 嘅先例：

Alessio 指出，LangChain 嘅成功本身就證明咗呢個觀點。LangChain 唔係一個模型——佢係一個 harness，一個將 LLM 連接到外部工具、數據庫同 API 嘅框架。LangChain 之所以贏，唔係因為佢嘅模型最好，而係因為佢提供咗 task-specialized harnesses——每個 harness 針對特定類型嘅任務（問答、Agent、RAG）進行優化。

「LangChain didn't win because it had the best model. It won because it had the best harness for every type of task.」 — Alessio Fanelli

Factory 推出「Design Mode」：

另一個相關動態係 Factory.ai 推出嘅 「design mode」——一個讓開發者以視覺化方式設計 agent 行為嘅工具。呢係 harness 抽象化嘅一個自然延伸：如果 harness 係產品，咁 harness 嘅設計工具就係產品嘅生產工具。

Provider-switching 對沖風險：

swyx 提出咗一個好實際嘅角度：provider-switching 正在成為 harness 設計嘅核心考慮。當你嘅 harness 可以無縫切換底層模型 provider——今日用 GPT-5.6，聽日用 Grok 4.5——你就從根本上對沖咗對單一模型 provider 嘅依賴。呢個能力對於 enterprise 客戶尤其重要，因為佢哋唔可以將整個 infrastructure 綁定喺一個供應商。

呢個趨勢解釋咗點解 Prime Intellect 將「bring your own harness」設為核心設計原則——唔係因為技術上優雅，而係因為市場上需要。你嘅 harness 必須同任何模型、任何 provider 相容，先至有長期價值。

五、Cost per Task——新嘅 eval 標準

呢節係全集最精彩嘅部分。swyx 同 Alessio 討論咗一個正在重塑行業認知嘅新指標：Cost per Task——唔係 token 價格、唔係 latency、唔係 MMLU 分數，而係「完成一個真實任務要花幾多錢」。

skirano 嘅 coding-agent index explorer：

呢個指標嘅推廣者係 skirano（一個著名嘅 AI 評估研究員），佢發布咗一個 coding-agent index explorer，比較唔同模型組合完成實際編程任務嘅成本。最引人注目嘅發現係 Terra Max 同 Fable 5 Max 嘅比較。

Cognition / Devin Fusion 改用 Fable 5：

呢個趨勢嘅一個具體案例係 Cognition 嘅 Devin Fusion——原本基於 Opus 4.8 嘅 coding agent，近期轉用 Fable 5 作為主力模型。原因好直接：per-task 成本低於 Opus 4.8，儘管 Fable 5 嘅 token 價格更高。呢個 counterintuitive 嘅發現解釋咗點解 token 價格唔再係好嘅比較指標。

Artificial Analysis 嘅觀點：

知名分析機構 Artificial Analysis 最近嘅報告總結得好直接：token 價格不再說明問題。兩個模型嘅 $/M token 可能差 5 倍，但因為更貴嘅模型需要更少嘅 token 來完成同一任務（更準確->更少 debug->更少 iteration），最終 per-task 成本可能更低。

Arena Agent Leaderboard 嘅衝擊：

呢個新嘅評估範式已經開始影響公開排行榜。最新嘅 Arena Agent Leaderboard 顯示：

  • GPT-5.6 Sol 排第 2 名
  • Grok 4.5 跌至第 13 名

呢個排名變化嘅背景係 Grok 4.5 雖然 token 價格極低（靠規模壓低成本），但喺 per-task 效率上唔及 Fable 5 或者 GPT-5.6 Sol——因為佢生成更多無效代碼、需要更多 iteration。呢個係 token 價格 vs. per-task 成本分歧嘅經典案例。

swyx 總結得精闢：

「We're entering the era where the metric that matters is 'can it finish the job, and how much does that finish actually cost?'. Everything else is a proxy.」 — swyx

六、Grok Build 隱私風波

Codex 同 Verifiers 嘅好消息背後，係一個令成個行業尷尬嘅隱私事件。Grok Build——xAI 推出嘅 coding agent——被 IntCyberDigest 指控存在嚴重嘅隱私問題。

指控內容：

IntCyberDigest 發表嘅報告稱，Grok Build 將用戶嘅完整倉庫上傳至 Google Cloud bucket。呢個唔係 metadata 或者摘要——據稱係完整嘅源代碼庫、包括 .env 文件、API keys 同敏感配置。

「Full repos being uploaded to a Google Cloud bucket, including .env files with API keys. This is a breach of basic trust.」 — IntCyberDigest 報告

xAI 嘅回應：

xAI 團隊（以 ZDR 為代表）迅速回應，聲明：

  • 設有 ZDR（Zero-Day Response）團隊處理呢類報告
  • trace/code 唔會長期保留
  • 用戶可以使用 /privacy 命令控制數據留存

但呢個回應並未平息社區嘅擔憂。關鍵問題有幾個：

  1. Default behavior 問題： 如果用戶唔知道 /privacy 命令嘅存在，默認行為係上傳完整倉庫——呢個設計本身就有問題
  2. 先前數據： 喺呢單事件曝光之前上傳嘅數據會點處理？xAI 冇明確交代回溯性清除政策
  3. 披露規範： IntCyberDigest 指控 xAI 喺隱私政策中未充分披露上傳行為嘅範圍

Arav Srinivas（Perplexity CEO）嘅觀點：

一個有趣嘅反應來自 Perplexity CEO Arav Srinivas。佢公開表示，ZDR（Zero-Day Response）嘅可用性 係 Perplexity 決定集成 Grok 4.5 嘅原因之一。呢個講法暗示，企業客戶喺選擇模型供應商時，安全響應能力同模型能力一樣重要——甚至更加重要。

「ZDR availability was one of the reasons we integrated Grok 4.5. You need to know there's a team that can fix things within hours, not weeks.」 — Arav Srinivas, Perplexity CEO

長遠影響：

swyx 同 Alessio 認為呢單事件會加速一個已經出現嘅趨勢：enterprise 對 open-source / self-hosted agent 嘅偏好增長。當你唔可以信任雲端 coding agent 唔會上傳你嘅源代碼，自我託管就變得更加有吸引力——即使犧牲一啲模型能力。呢個趨勢同 Verifiers v1 嘅開源策略完美互補。

七、其他關鍵動態

除咗上述主要話題，節目仲涵蓋咗其他幾個值得留意嘅動態：

Transformers ↔ vLLM 原生集成（Clément Delangue）：

Hugging Face CEO Clément Delangue 宣布咗 Transformers 庫同 vLLM 嘅原生集成。呢個意味住用 Transformers 訓練嘅模型可以直接 export 到 vLLM 格式，無需轉換步驟。呢個會大幅簡化從 research 到 production 嘅流程。

Continual Learning 重新成為一等問題：

隨住模型越嚟越大、部署越嚟越廣泛，「點樣喺部署後繼續學習」再次成為熱門研究方向。swyx 指出，過去幾年業界嘅注意力都喺 scaling laws 同 alignment，但而家愈來愈多人認識到——一個模型出廠之後就唔會變，呢個假設喺真實世界中唔成立。

LLM「睡眠與做夢」——sleep phase + dreaming phase：

一個有趣嘅研究方向係將生物學習中嘅「睡眠」（鞏固記憶）同「做夢」（探索性重播）引入 LLM 訓練。呢個類比令人聯想到人類點樣喺睡眠中整合日間經歷——AI 可能需要類似嘅機制嚟避免災難性遺忘（catastrophic forgetting）同提升泛化能力。

Rich Sutton Oak Lab 創業：追求動物級智能：

強化學習之父 Rich Sutton 宣布成立 Oak Lab，目標係追求 animal-level intelligence——唔係人類級通用人工智能，而係同動物類似嘅、喺物理世界中高效學習同適應嘅智能體。呢個方向嘅哲學同主流的「scale->AGI」路徑截然不同，更接近 Sutton 一直提倡嘅「The Bitter Lesson」——相信簡單嘅學習算法喺足夠計算下會贏。

ByteDance UniVR-34B：從視覺演示學習：

字節跳動發布咗 UniVR-34B，一個可以從視覺演示中直接學習嘅模型。同傳統嘅 text-based instruction following 不同，UniVR-34B 可以觀看人類演示（例如使用一個軟件界面或者組裝一個物體）然後模仿。呢個方向如果成熟，會打開 vast 嘅新應用場景——特別係 robotics 同自動化。

「Watching a human do something once and then doing it yourself — that's the dream. UniVR-34B isn't there yet, but the direction is clear.」 — Alessio Fanelli

Anthropic 300K+ 對話嘅價值觀表達研究：

Anthropic 發表咗一項大規模研究，分析咗 300,000+ 條對話中用戶同 AI 嘅價值觀表達模式。研究發現用戶傾向於喺對話中唔自覺地對 AI 進行「價值觀校準」——即係通過反饋同糾正嚟引導 AI 嘅輸出。呢個發現對 alignment 研究有重要意義：佢暗示咗一種分散式嘅「價值觀學習」機制可能喺日常使用中已經發生。

八、我的洞察

呢期 AINews 表面上係一集例牌新聞摘要，但實際上，swyx 同 Alessio 勾勒咗 AI 行業一個深層次轉型。以下係我綜合節目討論加上自己分析得出嘅五個核心洞察：

🔍 五大結構性洞察

  1. Codex 已經不僅是 coding agent——佢係 ChatGPT 嘅「內臟移植」

    6 個月 10x 增長，24.5 小時 +1M 用戶——呢啲數字嘅背後唔只係一個好產品。Codex 正在將「對話式編碼」變成開發者嘅默認工作流。如果 700 萬開發者每日用 Codex，佢哋產生嘅 token trace、feedback 數據同使用模式，本身就係一個巨大嘅數據 moat。Codex 唔係喺搶 coding agent 市場——佢係喺重新定義「點樣寫 code」。而呢個定義權一旦被 OpenAI 拿穩，其他 coding agent 就只能喺夾縫中生存。

  2. Cost per Task 取代 token pricing 係行業成熟嘅信號

    呢個轉變嘅意義遠超技術層面。Token 價格係一個「投入導向」嘅指標——我話俾你聽我嘅模型每一千個 token 要幾多錢。Cost per Task 係一個「產出導向」嘅指標——我話俾你聽完成你嘅工作要幾多錢。呢個轉變標誌住行業由「賣算力」走向「賣結果」。對於用戶嚟講，呢個係福音——你唔需要關心模型背後嘅 token 消耗細節。對於模型供應商嚟講，呢個係殘酷嘅——當用戶只關心 per-task cost，模型嘅品牌溢價會消失，競爭會變成純粹嘅效率競賽。

  3. 模型商品化後，價值往 infrastructure 轉移

    Prime Intellect 嘅 Verifiers v1、Harness 即產品趨勢、LangChain 嘅先例——呢三件事指向同一個方向：模型正在變成 commodity，但評估/部署/RL 基礎設施正在變成真正的護城河。Verifiers v1 嘅 message DAG 技術同「bring your own harness」設計，唔係 random 嘅工程決定——佢哋係對「模型會越來越便宜、越來越容易被替代」呢個未來嘅有意識回應。如果你嘅 infrastructure 可以支援任何模型、任何 task，你就永遠唔會被鎖死。

  4. Grok Build 隱私事件會加速 enterprise 對 open-source / self-hosted agent 嘅偏好

    呢個唔係一個孤立事件——佢係 cloud AI 隱私問題嘅典型代表。當 coding agent 需要讀取你嘅完整源代碼嚟完成任務，數據泄露唔係「可能性」，而係「時間問題」。嗰啲話「我哋唔會儲存你嘅代碼」嘅供應商，實際上面對嘅係一個可信度問題：用戶點樣驗證呢個承諾？呢個困境自然會推動企業走向 self-hosted 方案。同樣嘅邏輯適用於 Prime Intellect Verifiers v1 呢類開源框架——你可以自己 host 嘅時候，唔需要信任任何第三方。

  5. 「不寫代碼嘅 AI」反而更值錢——呢個 paradox 話俾我哋聽 AI 行業正在重新定義效率

    Fable 5 喺 81% 場景中唔直接寫代碼，但 per-task 成本最低。——呢個發現挑戰咗「寫 code 先係能力」嘅直覺。實際上，最值錢嘅 AI 行為可能係「規劃」、「驗證」、「推理決策」而唔係「生成代碼」。呢個見解對產品設計有深遠影響：與其追求「生成更多代碼嘅更快 AI」，不如追求「需要更少輸出就解決問題嘅更聰明 AI」。呢個轉變會影響從 prompt engineering 到 model architecture 嘅所有層面。
