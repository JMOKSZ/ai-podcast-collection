🏗️ Nathan 的個人AI架構

  • Claude Code 實例 — 包含1GB資料庫，編碼了Nathan 5年的完整數位歷史（郵件、日誌、筆記、項目文件）。

  • 代理層級設計（Agent Hierarchy） — 多層級代理結構，不同agent承擔不同職責與權限範圍。下層agent匯報給上層，最終由人類持有最終決策權。

🔐 安全架構與邊界

  • Miessler 著重審計了 信任邊界（Trust Boundary）：AI agent能訪問什麼、不能訪問什麼。
  • 強調 最小權限原則 在agent系統中的應用 — 每個agent只獲得完成任務所需的最少數據和工具訪問權。
  • 討論了 agent間通信的安全：agent相互通訊時的信息洩露風險。
  • 「如果一個agent能訪問我的郵件，它能給我發送郵件嗎？它能刪除郵件嗎？它能讀取附件嗎？這些都是不同的權限級別」— Miessler

💡 核心論點

  • 「Bitter Lesson Engineering」（苦澀教訓工程學）：借鑑Rich Sutton的名篇，Miessler認為設計AI系統時應接受「算力規模化最終會贏」的苦澀教訓。與其設計精巧的規則系統，不如構建能隨模型能力提升而自然增強的架構。
  • 「與AI分享你的理想狀態」：告訴AI你的理想生活狀態（不是任務清單，而是你希望世界如何運轉），AI會據此主動提出更貼心的幫助。
  • PAI 框架的核心：個人AI基礎設施不是一個單一模型或工具，而是一個協調多個AI agent圍繞人類目標的生態系統。關鍵組件包括 TELOS（目的定義）、分層記憶系統、agent編排層。
  • AI-人類互動的社會規範：當AI agent代表人類行事時，何時需要披露「這是AI在代表我」？目前缺乏社會共識。

🐛 多輪RL訓練的「分詞Bug」

  • 核心問題：解碼模型輸出 → 解析工具調用 → 重新分詞更新後的對話。重新分詞會改變token序列，導致梯度應用到模型從未實際採樣過的序列上。
  • 修復方案：嚴格執行「Token-In, Token-Out」規則：永遠不對已採樣的token重新編碼；在多次輪轉中保持單一token緩衝區。
  • 更廣義的教訓（@johnschulman2）：渲染器（Renderer）是消息與token之間的基礎設施，失效模式涵蓋訓練/測試不匹配、緩存效率低下、提示注入風險。
